# Traveler Bus Booking — Live Verification Matrix

**Verification date:** 2026-07-10
**Method:** No browser-automation tool was available in this environment. Every row below was verified by one of two honest means only:
1. **Verified via curl** — a real HTTP call against the running backend (`http://localhost:8080`, Spring Boot, H2 in-memory, seed data from `backend/src/main/resources/seed_data.sql`), with the actual observed JSON response quoted or summarized.
2. **Verified via automated test** — pointing at the exact spec file and `it(...)` test name (from Phases 1–5) that exercises that behavior against a component/service in isolation.

Where neither was possible (genuine visual/click-through browser interaction — e.g. actually seeing a rendered QR code image, physically clicking through the seat grid in two browser windows), the row is honestly marked **"Requires manual browser verification — not performed in this environment."**

**Test identity used:** seeded ROLE_TRAVELER `traveler@travelease.com` / `Traveler@123` (`userId 55555555-5555-5555-5555-555555555555`), logged in via `POST /api/auth/login`. A second traveler (`verify-second@travelease.test`, registered live via `POST /api/auth/register`) was used to prove the row-3 seat-lock conflict is a genuine two-user race, not a mock.

Both servers were running throughout: backend on `:8080` (`./mvnw.cmd spring-boot:run`), frontend dev server on `:4200` (`npx ng serve`, SSR-enabled).

## Honest summary of gaps found during verification (not fixed — validation-only phase)

1. **Row 12 (Modify booking):** `PUT /api/bookings/modify` is a fully working, real backend endpoint (proved live), but `booking-detail.html`'s "Modify" button has no click handler and no passenger-edit form exists anywhere in `features/bus-booking/`. This is a genuine frontend gap, not a browser-tooling limitation — flagged for whoever picks up the next increment, not fixed here per the "validation-only, no new features" constraint of this phase.
2. **Row 5 (LADIES seat):** cannot be live-curled because zero `LADIES`-type seats exist in current `seed_data.sql`; only verified via the isolated component test and source reading.
3. **Row 19 (Trip lifecycle gate) and Row 20 (New Booking CTA):** backend/route-level facts were proved live, but the specific frontend assertions (buttons hidden on a COMPLETED trip; CTA `routerLink` target) are not covered by any existing automated test found in this codebase, and no browser tool was available to click through them manually.
